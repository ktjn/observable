# Admin/Members Domain: Modelable Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add `models/admin.mdl` defining `admin.Member@1`, generate its TypeScript artifact, re-export it as `MemberRecord` from `apps/frontend/src/api/admin-members.ts` (deriving `TenantRole` from it), and add a lineage doc comment to the Rust `MemberRecord` — with no other Rust changes, plus a new Phase 1 backlog item documenting the `timestamp` -> Rust gap.

**Architecture:** This is a TS-only migration (Phase 3 step 3.4, scope "Option B" from the approved design). `MemberListResponse`, `AddMemberRequest`, `UpdateRoleRequest` stay hand-written — see `docs/superpowers/specs/2026-06-14-admin-members-modelable-migration-design.md` for the full rationale (modelable's `timestamp` primitive emits as Rust `String`, which would break `MemberRecord`'s `sqlx::FromRow` mapping of `joined_at: chrono::DateTime<Utc>`).

**Tech Stack:** modelable 0.4.0 (pinned in `models/requirements.txt`, no bump needed), TypeScript/Vite/Vitest frontend, Rust/Axum/sqlx backend.
