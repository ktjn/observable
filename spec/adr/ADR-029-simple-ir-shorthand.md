# ADR-029: Simple IR Shorthand & Deterministic Fallback

**Date:** 2026-05-03
**Status:** Proposed
**Authors:** Gemini CLI
**Deciders:** Project Stakeholders

## Context

Observable's Natural Language Query (NLQ) pipeline (as defined in [ADR-021](adr/ADR-021-nl-query-layer.md)) relies on an LLM to translate natural language into a structured Intermediate Representation (NlqIr). With the introduction of IR-driven signal pages, every query now carries a `base_ir` defining its context (e.g., Log Search context, Infrastructure context).

While the LLM provides powerful reasoning, it introduces latency (1-3s), depends on external availability, and can be non-deterministic for simple tasks.

## Decision

We will introduce a **Simple IR Shorthand** that allows the backend to perform deterministic query execution without calling an LLM. This shorthand serves two purposes:
1.  **Graceful Degradation**: A fallback when the LLM is unavailable (503).
2.  **Power User Bypass**: A low-latency path for users who know exactly what they want to search.

### Deterministic Fallback Logic

When a query is received, the backend will attempt a "Naive IR" generation if:
- The LLM provider is not configured or is unreachable (503).
- The user explicitly bypasses the LLM using a syntax prefix (`/`).

The Naive IR is generated by:
1.  Starting with the `base_ir` provided by the frontend.
2.  Parsing the `question` string using the shorthand syntax below.
3.  Merging the parsed values into the `base_ir` using the existing `merge_irs()` logic.

### Simple IR Shorthand Syntax

The shorthand uses a simple key-value and prefix notation:

| Syntax | Target Field | Example | Result |
|---|---|---|---|
| `m:<name>` | `metric` | `m:http_requests` | `metric = "http_requests"` |
| `f:<field>:<val>` | `filters` | `f:service:checkout` | `{field: "service", op: "=", value: "checkout"}` |
| `<field>:<val>` | `filters` | `env:prod` | `{field: "env", op: "=", value: "prod"}` (shorthand for `f:`) |
| `op:<type>` | `operation` | `op:topk` | `operation = "topk"` |
| `"quoted text"` | `query` | `"timeout error"` | `query = "timeout error"` |
| `unquoted text` | `query` | `checkout` | Appended to `query` |

**Prefixes:**
- `/` : Explicitly bypass LLM and use Shorthand.
- `!` : Reserved for "Force LLM" (even if auto-fallback is enabled).

### Examples

| User Input | Interpretation |
|---|---|
| `/error` | `query: "error"` (within current page context) |
| `/m:request_latency service:checkout p99` | `metric: "request_latency"`, `filters: [service=checkout]`, `query: "p99"` |
| `/op:catalog service_name` | `operation: "catalog"`, `catalog_field: "service_name"` |

## Consequences

**Easier:**
- **Zero Latency**: Direct commands bypass the LLM roundtrip, providing instant feedback.
- **Reliability**: Basic search/filter functionality works even if the LLM provider is down.
- **Predictability**: Power users get exactly what they typed without LLM "hallucinations" or reframing.
- **Local-First**: Enables full development and testing of the query execution layer without internet/API keys.

**Harder:**
- The backend needs a simple parser for the shorthand (likely a regex-based tokeniser).
- Users need to learn the shorthand (though it follows common CLI/search patterns).

**Constrained:**
- The shorthand is intentionally limited; complex reasoning (e.g., "compare X with Y") still requires the LLM.
- `base_ir` is required for shorthand to be effective, as it provides the signal context.
