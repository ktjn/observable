# Modelable Pilot: Generated SpanRow/SpanEventRow Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the hand-written `SpanRow`/`SpanEventRow` ClickHouse row structs in `libs/domain/src/span.rs` with modelable-generated types (`TracingSpanRowV1`/`TracingSpanEventRowV1`), as type aliases, with zero change to the public API or wire format.

**Architecture:** Commit the four Rust files `modelable compile` produces for the `tracing` domain into a new `libs/domain/src/generated/tracing/` module (private to the crate). `SpanRow`/`SpanEventRow` become `pub type` aliases to the generated `TracingSpanRowV1`/`TracingSpanEventRowV1` structs. The hand-written `Span`/`SpanEvent`/`SpanKind`/`StatusCode` types and the `From`/`Into` conversion *logic* between them are untouched — only the Row struct *definitions* (and their derives) become generated. One real wire-affecting change is required: the generated `TracingSpanRowV1` moves `parent_span_id` to the last field, so `SELECT_COLS` in `services/query-api/src/traces.rs` must be reordered to match (RowBinary SELECT decoding is positional).

**Tech Stack:** Rust, `clickhouse` crate v0.15 (`Row` derive), `serde`, modelable v0.2.1 (Python CLI, used only to regenerate the committed files — not a Rust build dependency).

### Task 4: Fix `SELECT_COLS` field order for the new `TracingSpanRowV1` layout

**Files:**
- Modify: `services/query-api/src/traces.rs`
- Modify: `services/storage-writer/src/spans.rs`

**Why:** `TracingSpanRowV1` (Task 2, Step 4) declares `parent_span_id` as its *last* field, whereas the old hand-written `SpanRow` had it 4th. `clickhouse` crate v0.15 inserts by column name (order-independent), but `fetch_all::<SpanRow>()` decodes ClickHouse RowBinary results *positionally* against the columns named in the `SELECT` statement. `SELECT_COLS` must list columns in the same order as the struct fields.
