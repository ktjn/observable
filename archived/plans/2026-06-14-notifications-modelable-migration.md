# Notifications Domain: Modelable Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add `models/notifications.mdl` defining `notifications.NotificationChannel@1`, generate its TypeScript artifact, and re-export it (aliased to `NotificationChannelItem`) from `apps/frontend/src/api/notifications.ts`, replacing the hand-written interface â€” with no Rust changes beyond a lineage doc comment.

**Architecture:** This is a TS-only migration (Phase 3 step 3.3, scope "Option B" from the approved design). `NotificationChannelItem` (Postgres `db` row), `CreateChannelRequest`, `NotificationChannelType` (Rust enum), and `NotificationChannelConfig` (TS type) stay hand-written â€” see `docs/superpowers/specs/2026-06-14-notifications-modelable-migration-design.md` for the full rationale (modelable's Rust emitter still emits `enum(...)` as `String`, and `json` maps to TS `unknown`).

**Tech Stack:** modelable 0.4.0 (pinned in `models/requirements.txt`, no bump needed), TypeScript/Vite/Vitest frontend, Rust/Axum/sqlx backend.
